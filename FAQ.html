I am using qsub to execute my job.

I am using qsub to execute my job. The status of my job is always C, not Q or R. Might there be something wrong with my PBS script?

A state of 'C' means completed - the job went from Q (queued), to R (running), to C (completed), assuming the job wasn't deleted. Make sure you have 

   #PBS -j oe

In your PBS. After submitting your job, look or the output files/or some error log. Double check your source code, make sure it has an expected output of some sort.

Disk Quota Exceed Messege

When I use packages on Hammer to calculate something, it constantly ended with Disk Quota Exceed. Does that mean I should increase the number of cores? Or is there any other problem?

Most likely you have exceeded the hard drive disk space assigned to you. The terminal commmand mmlsquota will give you your disk usage and we suggest that you always keep some space available to avoid weird behavious! You have several directories available to you to store files in: your home directory, your work directory, and your scratch directory. Your home directory has a disk quota (storage limit) of 8 GB. Your work directory has a disk quota of 64 GB. Your scratch directory does not have a limit on the amount of storage used, but files that haven't been accessed in 30 days will be automatically deleted. You can access your work directory either through the symbolic link called 'work' in your home directory or at /gpfs/work/your_PSU_user_ID.

Real time team debugging on our clusters

I've got a few teammates working on MATLab programs for the first time on the Hammer server. We're using these programs to analyze lots of different types of data in our lab. Most of the bugs that they run into are quick fixes, so it would be a huge time saver if another teammate could log right into an existing session realtime and help them debug.
I ask because without logging-in directly to the session, that other teammate would have to then replicate some unlikely error from a saved file in a shared directory.
I'm thinking that this situation is different from the Group Account solution RCC offers.
The goal here is to give my research team the ability to open the session of another team member and debug a task/project or pick up where another left off. (I'm not sure if this means realtime collaboration, or even what the implementation limits are.)
Is a group account an appropriate solution here, or will this on permit shared access to directories and not actual sessions?
What you want to do is different from the group account.

You can do shared sessions if you connect to Hammer using Exceed onDemand:
http://rcc.its.psu.edu/user_guides/remote_display/exceed_on_demand The basic flow of operations is:
1. The person sharing the session opens up an Exceed onDemand session, sets their share list, and enables sharing.

2. The person wanting to join the session then opens Exceed onDemand, authenticates with Hammer, does *not* open a new session but instead goes into the shared sessions listing and joins the shared session.

This is something you should be *very* careful with though. Anything done within the shared session will be done under the credentials of the person sharing the session. If Innocent Mike shares his session with Bad Joe and Bad Joe thinks it'd be funny to hack into a FBI system using Innocent Mike's session, the feds are going to be coming to see Innocent Mike because it'll look like he did it. There's a lot of potential for abuse here and I'd recommend that if someone is sharing their session, they don't leave it shared and unattended. To find specific steps on creating and participating in a shared session, you can open Exceed onDemand, select "Contents" under the "Help" menu, and click on the "Sharing X Sessions" subject on the left-hand side Content listing.

What is a group directory?

A group directory will allow the sharing of files since all members of the group can store files in the same location. It won't necessarily allow the sharing of some session. It would allow one group member to open some saved file that another group member created and then continue work from the state saved in that file.

Please send me instructions on how to access Gauss software from the LION-XG cluster.

I have access to the HPC cluster, and am interested in using the Gauss program for running some analyses.

You can add Gauss to you runtime environment by running 'module load gauss'. Once you run 'module load gauss', you can launch Gauss via the commands 'tgauss' or 'gauss'.

tgauss is the command-line version of the command. gauss is the GUI version of the command. If you're running in batch on Lion-XG, you'll want to use tgauss since Lion-XG is a batch computing cluster. General information on running on Lion-XG can be found at: http://rcc.its.psu.edu/user_guides/systems
You will need to submit your jobs to the PBS job queuing system on Lion-XG for them to run on the Lion-XG compute nodes. Jobs are submitted to PBS through the use of a PBS script. An example for Gauss would be: 


   #PBS -l nodes=1:ppn=1
   #PBS -l walltime=1:00:00
   #PBS -l pmem=1gb
   #PBS -q lionxg-smeal
   #PBS -j oe

   cd /path/to/guass/input/file
   module load gauss
   tgauss -b input.e

PBS and PBS scripts are covered in the User Guides linked to above. Gauss examples can be found in /usr/global/gauss/13.0.5/examples. The file 'circular.e' works well for a quick test to see if a PBS script works.

Alternatively, if you want to use the GUI version of Gauss, we have an interactive cluster called Hammer that you can use. I can send you information about that separately if this is a path you want to pursue.

Is there any policy which restricts the number of processes that can be run on the Cyberstar system

I would like to know if there is any policy which restricts the number of processes that can be run on a limited set of nodes/cores on the Cyberstar system, submitted through pbs? For e.g., Can one launch 512 processes on 64 cores [which theoretically should be okay], or if there is a policy in place which will kill such a job?

There is no policy in place where a job that's using more processes than requested cores will be automatically killed. However, if you're going to run more processes than the number of cores that you requested, you will need to make sure that you're not using resource that's been allocated to someone else. For example, if you request one processor core per node and then run 24 processes on each node, your 24 processes are going to fight with other people's processes on those nodes for CPU time. In this case, we'd kill your job because you're using more resource than you requested. The best way to avoid this would be to request all processor cores on the nodes you are requesting since in that case there would not be any processes from other users on those nodes to worry about.

Asking for nodes on the Cyberstart cluster

I'm running a couple of huge jobs on the CyberSTAR, If I have to run say 512 processes on 64 cores, I should better request for nodes=8:ppn=8 rather than just nodes=64. I hope this is the exact way to go.

On CyberSTAR, you'll have to be a bit more specific than just nodes=8:ppn=8 since CyberSTAR has three different node types in it.

What you probably want is:
nodes=8:nehalem:ppn=8

Since the Nehalem nodes are the ones with eight cores per node ( there are also five Nehalem nodes still reserved for SSCL). If you don't specify the 'nehalem' feature, your job could also run on the Shanghai or Westmere nodes.

I cannot open NASTRAN on the hammer cluster.

I cannot open NASTRAN on the hammer cluster. I load the module "module load nastran", after to run nastran I input the command "nastran mynastran_model.bdf" I get an error saying the file does not exist. Is this the correct procedure to run nastran?

Since NASTRAN is a solver only the command "nastran mynastran_model.bdf" instructs NASTRAN to solve the model named mynastran_model.bdf. If you do not have any model named mynastran_model.bdf in your current directory you will get the error message saying the the file does not exist. To set up a model you can solve with NASTRAN, you will have to use PATRAN, which is the pre and post processor of NASTRAN.
To use PATRAN, just load the patran module ("module load patran"), then run the command "patran". You can find a tutorial on NASTRAN/PATRAN at
http://rcc.its.psu.edu/resources/software/nastran/Nastran-Tutorial.pdf
